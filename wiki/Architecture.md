# Architecture

Single HTML file (`index.html`): CSS design tokens in `:root`, markup (shell + 8 views + inspector), ~970 lines of vanilla JS.

## Shell
- Left icon rail → `switchView(v)`; views are stacked `<section class="view">` toggled by `.is-active`
- Top bar → live stats, posture pill, analyst/clearance switcher, UTC clock
- Fixed classification banner above the shell when markings are on (`body.marked`)
- Right inspector drawer → `selectEntity(id)` from any surface; relationship rows pivot to neighbors. `switchView()` force-closes it when navigating to any view other than Constellation (fixed 2026-07-09 — it used to stay open and show stale content over unrelated views since it's a global overlay, not scoped to `.view`)

## Views
| View | id | Purpose |
|---|---|---|
| Command Center | `view-overview` | metrics, gauge, severity bars, live feed, top actors |
| Constellation | `view-graph` | canvas force graph |
| Threat Actors | `view-actors` | filterable actor list + editable profile (image gallery, meta, local-neighborhood constellation preview) |
| ATT&CK Matrix | `view-matrix` | observed-technique highlighting |
| Timeline | `view-timeline` | event chronology |
| Ingest | `view-ingest` | drop/paste/parse + per-batch options |
| Search & Alerts | `view-alerts` | query, saved searches, alert log |
| Pipelines & Settings | `view-pipes` | auto-ingest sources, markings, translation |

## Graph engine
`requestAnimationFrame` loop → `physics()` (pairwise repulsion 2600/d², springs rest-length 110 k=0.012, centering 0.0016, damping 0.86, velocity clamp ±8) → `draw()` (grid, edges, pulse particles, nodes with severity rings and redaction). Camera `{x,y,k}` with cursor-anchored wheel zoom (0.25–3×). O(n²) — fine to ~300 nodes; quadtree is the scale fix (see [[Roadmap]]).

The force loop's per-tick body lives in `physicsStep()`; `physics()` just gates it on the `frozen` toggle. `organizeLayout()` (the graph controls' "Organize" button) calls `physicsStep()` 180 times synchronously to fast-forward the simulation to a settled state on demand, bypassing `frozen`.

Edges render via `relColor(relation)` — a deterministic hash of the link's `relation` string onto the `TYPES`/`SEV` palette (`hexToRgba()` applies opacity) — with a matching "Relationship Lines" legend section (expandable past 20 via `legendRelExpanded`, since large ingested graphs can have dozens of distinct relation strings). The inspector's Relationships section colors each relation's text the same way, so the two surfaces read consistently.

`linkAt(sx,sy)` mirrors `nodeAt()` — point-to-segment distance in world space — to hit-test edges. Nodes win when both are under the cursor (checked first in both the hover and click handlers). Hovering an edge shows a floating tooltip (`#edgeTip`, positioned next to the cursor, border-colored via `relColor`); clicking one shows a toast (`source — relation — target`) instead of closing the inspector. The hovered edge also gets the same brighter/thicker treatment as a selected node's edges.

**Per-node pinning and hiding** (both view-state only, not exported — same treatment as `x`/`y`/`vx`/`vy`): dragging a node further than 4px sets `entity.pinned=true` (mouse and touch); `physicsStep()` zeroes velocity and skips integration for pinned nodes, so they act as fixed anchors that still exert repulsion/spring forces on everything else. Toggle from the inspector's "Pin in place" button; pinned nodes get a dashed ring in `draw()`. `entity.hidden=true` (inspector "Hide node") excludes an entity from `visibleEntities()` and from any edge touching it (`physicsStep()`, `draw()`); the graph controls' "Unhide all" button (auto-shown/hidden via `updateUnhideBtn()`, called from `render()`) clears every `hidden` flag at once — there's no per-node unhide UI, only clear-all.

**Actor focus**: the Actors panel (`#actorsPanel`, same `.legend` styling as the entity-type/relationship legend, stacked above it in `.graph-hud`) lists every `type:'actor'` entity with its connected-entity count, auto-shown once 2+ actors exist. `actorComponent(id)` does an undirected BFS over `state.links` to find everything reachable from that actor; `focusActor(id)` sets `entity.hidden` for the whole graph based on component membership — reusing the same hide mechanism above, not a separate filter system. `focusedActorId` (null = "Show all") drives the active-row highlight; "Unhide all" also clears it so the two controls can't disagree. Boot/ingest behavior is unaffected — this is opt-in decluttering, never automatic.

**Group filter on the Actors panel** (added 2026-07-09, for when multiple campaigns are loaded and the flat actor list gets long): a `<select>` (`#actorsPanelGroupSel`, shown only when 2+ campaigns are present) listing every `campaign` entity that has at least one directly-linked actor. Picking one narrows which actor *rows* are listed (`actorsPanelGroupFilter` state, filters by direct campaign↔actor link membership) — it does not itself change what's focused in the graph; that's still a separate click on an individual actor row or "Show all". Deliberately decoupled: the dropdown declutters the list so you can find the actor you want, the row click still does the actual graph focus.

## Threat Actors view

A dedicated view (`view-actors`, `.actorsview` — fixed-width filterable list on the left via `renderActorsView()`, profile panel filling the rest via `renderActorProfile()`), separate from both the compact Actors panel above and the generic inspector drawer. `profileActorId` tracks which entity's profile is open (distinct from `selectedId` — the generic inspector's selection — and `focusedActorId` — the graph filter); redaction is enforced the same way as the inspector (`selectActorProfile()` refuses a redacted entity with a clearance toast; `applyUser()` closes an already-open profile if a clearance switch makes it redacted).

**Not actually actor-only, despite the name**: `selectActorProfile()`/`renderActorProfile()` never checked `e.type==='actor'` — they're generic entity-editing code that happened to only be *entered* from actor contexts at first. Since 2026-07-09 `campaign` entities are profiled through the exact same path (see Groups, below); the only place that assumed actor-specific styling was the avatar placeholder color (hard-coded `TYPES.actor.color`), now `TYPES[e.type].color`.

**Sidebar structure — Groups then Actors** (added 2026-07-09, the flat actor list stopped being useful once a large dataset like the Americans network — 51 actors, all showing an identical "60 connected" since that's the whole shared component, not each actor's own significance — was loaded): `renderActorsView()` renders a "Groups" section (every `campaign` entity with at least one directly-linked actor, member count shown) above the "Actors" section when any campaigns are present. Clicking a group both opens its own profile (`selectActorProfile(campaignId)` — see above, campaigns profile like actors now) *and* sets `actorsViewGroupFilter`, narrowing the Actors section below to that group's members (a "Show all groups" row clears it). Actor rows now sort by **direct** connection count (`directConnectionCount()` — a plain `state.links` filter) descending, not `actorComponent()`'s whole-component size — in a single large connected network every actor used to show the same number; direct-connection count actually differentiates Philip Jennings (24 direct ties) from a peripheral one-scene character (1).

Everything in the profile is editable and writes straight back to the entity, re-rendering on each change (label, severity via `<select>`, classification, tags, and free-form `meta` key/value pairs — add/edit/remove rows, generalizing the tag-chip add/remove pattern to arbitrary attributes). The header's action button is labeled with the entity's own name — "Show only *Label* in constellation" — and calls the existing `focusActor(e.id)` + `switchView('graph')`; the profile doesn't run its own full-graph rendering, it hands off to the real one.

**Profile images (gallery, not a single avatar)**: `entity.images` is an array; `handleImagesAdd(fileList, entity)` accepts multiple files at once (the file input has `multiple`), resizes each to ≤480px on the long edge via an off-screen `<canvas>`, and pushes a JPEG data URI (quality 0.82) per file — no server, no separate asset storage, round-trips through JSON export/import like every other field. `images[0]` is the avatar shown in the header/list; a thumbnail strip below the header (`.profile__gallery`) shows every image with a per-thumbnail delete button plus a dashed "+" tile that opens the same file picker. Clicking the avatar also opens it. A placeholder (first letter of the label, in the entity's own `TYPES` color) shows when `images` is empty.

Clicking a relationship in the profile behaves differently depending on what's on the other end: another actor *or campaign* swaps the profile to them (`selectActorProfile`, stays in this view); anything else jumps to the Constellation view and opens the generic inspector on it (`switchView('graph')` + `selectEntity()`). Same broadened check in the local-neighborhood preview's center-node click.

**Threat Group / Campaign section**: a profile section, shown above Classification when it applies, listing every `type:'campaign'` entity linked to the profiled actor (filtered from the same `rels` computation the Relationships section uses lower down). The row itself is clickable (`data-goto-actor`, same navigation as a Relationships row — opens the campaign's own profile) and separately has a "Show only this group" button (`focusActor(campaign.id)` + `switchView('graph')`, `ev.stopPropagation()` so clicking the button doesn't *also* trigger the row's profile-navigation). `focusActor()`/`actorComponent()` were already entity-agnostic BFS/hide logic despite the actor-centric naming, so no new graph-filtering code was needed to root the focus at a campaign instead of an actor. Redacted campaigns blur and drop both the row's click target and the button.

**Local-neighborhood constellation preview**: `.profile__previewPanel` — a sticky, non-scrolling panel filling the space to the right of the (narrower, `max-width:760px`) profile fields. Holds its own `<canvas id="profilePreview">`, redrawn by `drawProfilePreview()`. Deliberately *not* the whole connected component (that was the v0.8.2 design — unreadably dense for anything but a small actor, and unlabeled). Instead it's a radial 1-hop layout: the last entry of `previewTrail` (an array, reset to `[profileActorId]` whenever `selectActorProfile()` opens a different actor) sits at the center, its direct neighbors (one `state.links` filter) are placed in a circle around it, and **every** node is labeled — center in `--sans` bold above it, neighbors in `--mono` below/above depending on which half of the circle they land in to avoid overlapping the ring.

**Full breadcrumb trail** (redesigned 2026-07-09 — a single "« back to root" link wasn't enough once you're several hops deep and can't see the path you took): `#previewHead` renders every entry in `previewTrail` as a clickable crumb except the last (current, bold, non-clickable), separated by `›`. Clicking an earlier crumb truncates the trail to that point (`previewTrail = previewTrail.slice(0, i+1)`) and redraws. `rootCampaignFor(profileActorId)` prepends the actor's home campaign as a fixed first crumb *before* the trail — "the overall group this whole exploration is rooted in" — computed once against `profileActorId` (not whatever's currently centered), so it stays constant as you drill around. Clicking the campaign crumb jumps the trail to `[campaign.id]`, letting you explore outward from the group level; this reuses the same click-driven navigation as any other crumb rather than the separate "Threat Group / Campaign" section's full-graph-filter button, so the two stay conceptually distinct (local re-center vs. leaving the profile for the real graph).

**Pulse animation and edge hover** (added 2026-07-09, matching the main Constellation's feel): `tick()` calls `drawProfilePreview(true)` every frame when the Threat Actors view is active and a profile is open — the `animate` flag gates pulse spawn/advance (`previewPulses`, same shape and cadence as the main graph's `pulses`: random ~6%/frame spawn chance among the current neighbor edges, `t` advances 0.025/frame, expires at `t>=1`) so calls from click/crumb handlers can redraw without perturbing pulse timing. Pulses are cleared (`previewPulses=[]`) on every re-center — otherwise a pulse's `ni` index would point at the wrong edge once the neighbor list changes underneath it. Edge hover mirrors the main graph's `linkAt()`/`#edgeTip` pattern at small scale: `profilePreviewLinkAt(sx,sy)` does point-to-segment distance against the radial center↔neighbor positions (no world-to-screen conversion needed, they're already canvas pixels), a dedicated `#profileEdgeTip` tooltip (reuses the global `.edge-tip` CSS class, positioned relative to `.profile__previewPanel`'s `position:sticky` containing block) shows the relation on hover, and the hovered edge renders brighter/thicker (`previewHoverLink`) — nodes still win when both are under the cursor, same priority order as the main graph.

Click has two tiers, mirroring how you'd expect a local map to behave: clicking a **neighbor** pushes it onto `previewTrail` and redraws (pure local navigation, no view change); clicking the **center** node opens it for real (`selectActorProfile` if it's an actor and stays in this view, otherwise `switchView('graph')` + `selectEntity()`). Redacted nodes are drawn grey with a `REDACTED` label and are inert to clicks either way — including a defensive loop inside `drawProfilePreview()` that pops entries off the *end* of `previewTrail` while the resulting center is redacted (falling back toward the root, and bailing out entirely if even a single-entry trail is redacted) if the current focus becomes redacted out from under it, e.g. a clearance switch while drilled into a neighbor. Since positions here are computed radially rather than read from the live simulation, the preview no longer needs a per-frame redraw — it only redraws on open, on re-center/crumb-click, and on clearance change (`applyUser()` calls `drawProfilePreview()`/`updatePreviewHeader()` when the focused actor isn't itself newly redacted).

## Design language
Signals-intelligence observatory: indigo void, signal cyan accent, calibrated severity ramp (blue→yellow→orange→red), Space Grotesk for UI, JetBrains Mono strictly for data. Classification colors follow convention: U green, CUI purple, S red, TS orange.
